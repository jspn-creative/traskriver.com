# Milestone v2.0: Paywall Prereqs

**Status:** ✅ SHIPPED 2026-03-20
**Phases:** 01-04
**Total Plans:** 7

## Overview

Shipped signed URL streaming and all prerequisites for a paywall. The site now delivers a live stream behind Cloudflare Stream's signed URL protection, with immediate page shell rendering. Paywall enforcement is deferred to v3.0 pending client decision on payment model (one-time purchase vs. subscription).

### Phase 04: Signed URL Streaming

**Goal:** Restore stream playback with CF Signed URLs + server-side JWT signing + async page delivery.
**Depends on:** 03
**Plans:** 3 plans

Plans:

- [x] 04-01: Create CF Stream signing key provisioning script

## Milestone Summary

**Key Decisions:**

- Automatic auth bypass — defer paywall until client decides payment model
- Cloudflare Stream for RTSP ingestion — RTMPS push via ffmpeg, HLS delivery handled by CF
- `$env/dynamic/private` for all CF env vars — required for Workers edge runtime
- `error(404)` on dev-only endpoints — avoids leaking endpoint existence in production
- RS256 JWT via Web Crypto — no CF API calls per request, pure edge-compute signing
- Nested `<svelte:boundary>` pattern — scopes async loading to container, not full page

**Issues Resolved:**

- Local ffmpeg-to-HLS incompatible with Cloudflare Workers deployment → resolved with RTMPS push
- Plain HLS URLs returned 401 (CF "Require Signed URLs" enabled) → resolved with server-side JWT signing
- Full-page async boundary blocked header/sidebar rendering → resolved with nested boundary

**Issues Deferred:**

- Paywall enforcement (PAY-01, PAY-02, PAY-03) — pending client payment model decision
- 5-minute session timeout (SESS-01) — deferred to v3.0 paywall phase
- LiveViewerCount calls `getStreamInfo()` twice (potential duplicate fetch) — not confirmed, needs investigation

**Technical Debt Incurred:**

- Authentication is open (all visitors auto-authenticated) — temporary until v3.0 paywall ships
- No Stripe webhook handling yet — lifecycle management deferred to v3.0
