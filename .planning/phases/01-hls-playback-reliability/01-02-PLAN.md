  <action>
Two changes to `+page.svelte`:

**1. Tighten `onPlaybackError` to only transition from `viewing` (RC3 fix):**

The feedback loop happens because `onPlaybackError` transitions from `phase === 'live'` to `ended_confirming`. But `live` means "relay says live, HLS not yet playing" — a fatal HLS error during this phase is a startup failure, NOT a stream ending. The relay is still live, so polling will say "live" → back to `live` phase → loop.

**Before (lines 204-211):**

```typescript
const onPlaybackError = () => {
	log('onPlaybackError — resuming relay polling for end confirmation');
	streamError = true;
	if (phase === 'viewing' || phase === 'live') {
		phase = 'ended_confirming';
		polling = true;
	}
};
```

**After:**

```typescript
const onPlaybackError = () => {
	streamError = true;

	if (phase === 'viewing') {
		// Playback was established then failed — confirm if stream actually ended
		if (__DEV__) console.debug('[page] playback lost during viewing — confirming stream status');
		phase = 'ended_confirming';
		polling = true;
	} else if (phase === 'live') {
		// Fatal HLS error during startup (before playback established).
		// Stay in 'live' — VideoPlayer's last-resort remount will retry.
		// Do NOT transition to ended_confirming (prevents RC3 feedback loop).
		if (__DEV__) console.debug('[page] HLS fatal error during startup — staying in live phase');
	}
};
```

This breaks the feedback loop: when in `live` phase and a fatal error occurs, the page stays in `live`, and the VideoPlayer's last-resort single remount (from Plan 01) will retry after 10s. If that also fails, the user will see the stream attempting to connect, and can use the restart button.
