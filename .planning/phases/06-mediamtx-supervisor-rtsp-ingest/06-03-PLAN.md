**`packages/stream/src/index.ts`** — full replacement (Phase 5 boot grew to integrate Supervisor):

```ts
import { serve } from '@hono/node-server';
import { loadConfig } from './config.ts';
import { createLogger } from './logger.ts';
import { createApp } from './server.ts';
import { Supervisor } from './supervisor.ts';

const config = loadConfig();
const rootLog = createLogger({ level: config.LOG_LEVEL, nodeEnv: config.NODE_ENV });
const log = rootLog.child({ component: 'server' });

const supervisor = new Supervisor(config, rootLog.child({ component: 'supervisor' }));

const app = createApp({ getStatus: () => supervisor.getStatus() });

const server = serve({ fetch: app.fetch, port: config.PORT, hostname: '0.0.0.0' }, (info) =>
	log.info({ port: info.port }, 'stream service listening')
);

void supervisor.start().catch((err) => {
	log.error({ err }, 'supervisor failed to start');
	process.exit(1);
});

async function shutdown(signal: NodeJS.Signals) {
	log.info({ signal }, 'shutdown signal received');
	try {
		await supervisor.shutdown();
	} catch (err) {
		log.error({ err }, 'error during supervisor shutdown');
	}
	server.close((err) => {
		if (err) {
			log.error({ err }, 'error closing server');
			process.exit(1);
		}
		process.exit(0);
	});
}

process.on('SIGTERM', () => void shutdown('SIGTERM'));
process.on('SIGINT', () => void shutdown('SIGINT'));
```

Critical ordering rules (RESEARCH §Focus-Area Deep Dive #7 — non-negotiable):

**Boot order:**

1. `loadConfig()` — fail-fast on invalid env (existing Phase 5 behavior)
2. `createLogger(...)` — root pino instance with redact applied (Plan 01)
3. `new Supervisor(config, childLog)` — CONSTRUCT only, do NOT call `.start()` yet — `getStatus()` works immediately and returns `'starting'`
4. `createApp({ getStatus: () => supervisor.getStatus() })` — closure captures the supervisor reference
5. `serve(...)` — Hono begins listening; `/health` immediately answers `'starting'`
6. `void supervisor.start()` — fire-and-forget; spawn failure caught and treated as fatal

**Shutdown order (REVERSE of startup, per CONTEXT §Backoff + Lifecycle):**

1. Receive SIGTERM/SIGINT
2. `await supervisor.shutdown()` — blocks until MediaMTX child is gone (≤10s; Plan 02 guarantees this)
3. `server.close(callback)` — Hono drains in-flight `/health` requests
4. `process.exit(0)` (or 1 on close error)

If supervisor shutdown fails (try/catch logs error), STILL close the server and exit — do not leave Hono stranded.

**Why supervisor BEFORE server.close():**

- Operator-visible `/health` should keep responding (`'shuttingDown' → 'degraded'`) while MediaMTX is being torn down. Closing Hono first would 404 ops checks during the 10s shutdown window.
- If MediaMTX outlives Node (e.g. Node closes Hono and exits before SIGTERM lands on the child), journald shows a dangling "mediamtx still running" log that confuses operators.

**Wrapping rules:**

- `void supervisor.start().catch(...)` — explicit `void` because we don't await the promise; we want HTTP up before MediaMTX boot completes.
- `process.on('SIGTERM', () => void shutdown('SIGTERM'))` — wrap the async call in `void` because Node's signal listener type is `() => void`.
- `try/catch` around `await supervisor.shutdown()` so a supervisor error does not skip `server.close()`.

**No explicit return types** anywhere (user rule).

**No imports beyond:** `@hono/node-server`, `./config.ts`, `./logger.ts`, `./server.ts`, `./supervisor.ts`. No Bun, no node-fetch, no execa, no yaml lib.
